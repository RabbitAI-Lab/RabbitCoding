# 重构 Checkpoint/Rewind 交互为“点击编辑即回滚”模式

_来源：dc1bf71 → 717fd1f 提交周期内记录的编码计划——内容为规划时意图，实现可能滞后或有出入。_

**状态：** accepted

## 背景
原有的 Rewind 功能依赖独立的按钮和确认弹窗，交互路径较长且不够直观。用户希望更自然地通过“编辑历史消息并重发”来触发上下文回滚，以修正 Agent 的执行路径。

## 决策驱动
- 用户体验直观性（减少点击次数和认知负担）
- 交互流畅度（消除确认弹窗打断）
- 利用已有的 Sidecar 层 checkpoint 能力

## 备选方案
- **点击 User 消息填入输入框，发送时自动触发 rewind 并截断后续消息** — 优点：交互符合直觉（编辑即重做）；无需额外 UI 控件；自动清理无效的历史分支状态。；缺点：用户可能误触导致意外回滚（需通过仅在非运行态允许点击来缓解）；需要精确管理 sessionId 重置与消息截断逻辑。
- **保留独立的 RewindButton 和确认弹窗** _（已否决）_ — 优点：操作意图明确，防止误操作。；缺点：交互繁琐；打断了用户的思维流；UI 元素冗余。

## 决策
删除 `RewindButton.tsx`，改为在 `AgentMessage.tsx` 中使 User 消息可点击。点击后将消息文本填入 Sender 输入框并记录 `editingCheckpointRef`。在 `handleSubmit` 中检测该标记，若存在则异步调用 `agent.rewindFiles` 恢复文件状态，同步调用 `store.truncateFromMessage` 截断当前消息及之后的所有记录并重置 `sessionId`，从而以“新查询”的方式重新开始对话。

## 影响
显著简化了回滚操作的 UX，使其成为对话流的自然部分。但要求前端状态管理（`useWorkspaces`）必须精确处理消息截断与会话 ID 重置的原子性，否则可能导致上下文状态不一致。同时移除了相关的 i18n 文案以减少维护负担。
